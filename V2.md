Chess Arena V2 Plan
Vision
Chess Arena V2 should become a chess platform first, with optional competitive and Web3-powered modes layered on top. The core game experience should not require gambling, wallets, escrow, or tokens. Players who simply want to play chess should be able to match, play, finish, and build a rating history without touching blockchain features.

The funded parts of the platform should be explicit product modes:

General Games: free casual or rated chess games.
Challenges: funded head-to-head games where the winner takes the escrowed pot and draws refund both players.
Tournaments: organizer-created competitions where winners can receive NFTs, USDT, native tokens, points, or other configured rewards.
This separation makes the system safer, clearer, and easier to scale. The chess engine stays pure. Money movement stays isolated. Tournament orchestration becomes a layer above normal games instead of being mixed into move handling.

Primary V2 Goals
Refactor the backend from Django to FastAPI.
Separate game logic from transport, persistence, escrow, and settlement.
Support three game products: General Games, Challenges, and Tournaments.
Make wallets optional for general play.
Require escrow only for funded Challenges.
Support tournament rewards through a clear prize distribution system.
Replace single private-key settlement with multisig-governed settlement.
Fix the critical audit issues from V1.
Create a clean monorepo structure with one dependency system and one unified test suite.
Make make test, make lint, and make dev the standard local workflows.

Game Engine Rules
The chess engine should be pure Python and heavily tested.

Responsibilities:

validate legal moves using python-chess
apply moves
produce updated FEN
detect checkmate
detect stalemate
detect insufficient material
support draw offers
support resignation
support timeout
track move history
Clock responsibilities:

start when the game starts, not when the first move is made
charge the player whose clock was running
persist server-authoritative time
allow timeout claim without requiring the timed-out player to move
never trust client-side timer display as authority
Rating responsibilities:

calculate both players' new ratings from original pre-game ratings
update ratings exactly once per rated game
skip rating changes for unrated games
store enough audit data to explain rating changes later
Multisig Settlement Design
V1 used a single server-side private key as a judge. V2 should replace this with multisig-governed settlement.

Recommended approach:

Use a Safe/multisig wallet as the settlement authority.
The backend should not directly hold the final authority to move funds. Instead:

The game engine determines the result.
The backend creates a settlement request.
The backend prepares the settlement payload.
Multisig owners review and approve/execute the transaction.
The contract pays the winner or refunds players.
The backend verifies the settlement transaction and marks the record settled.
Contract authorization options:

Option A: only the multisig address can call settlement functions.
Option B: contract verifies EIP-1271 signatures from the Safe.
Recommended V2 path:

Start with Option A for simplicity.
Upgrade to Option B if off-chain signed settlement payloads become necessary.
Settlement payload should include:

chain id
contract address
challenge id or tournament id
game id
result
winner address if any
player addresses
asset address
amount
nonce
deadline
Contract V2 Requirements
The V2 contract should support Challenges first. Tournament prize contracts can be added after the challenge escrow is solid.

Challenge Escrow Contract
Required behavior:

create a challenge escrow

prevent game start until both deposits are present
settle winner payout
settle draw refunds
cancel/refund unfunded or expired challenges

Must remove or redesign:

claim_abandonment
The old V1 version lets either player claim the pot after 24 hours, regardless of the off-chain game state. V2 abandonment handling must be tied to a verified server result, timeout proof, multisig adjudication, or a safe cancellation/refund path.

Fixes From V1 Audit
V2 must explicitly fix these issues:

P0: Unsafe Abandonment Claim
V1 problem:

Either player can steal the full pot after 24 hours.
V2 fix:

Remove arbitrary pot claim.
Use safe expiry refunds before game start.
Use timeout/adjudication settlement after game start.
Require multisig-authorized settlement for disputed or abandoned games.
P0: Wagered Game Can Start Without Escrow
V1 problem:

WebSocket accepts moves if both players are assigned, even when deposits are missing.
V2 fix:

Challenge games only enter IN_PROGRESS after escrow is fully verified.
MoveService rejects moves unless the game status is IN_PROGRESS.
P1: Elo Re-applies On Later Saves
V1 problem:

Rating update guard is only in memory.
V2 fix:

Persist rating update records.
Use a transactionally enforced one-time rating update.
P1: Rating Uses Mutated Opponent Rating
V1 problem:

Black rating calculation uses white's already-updated rating.
V2 fix:

Calculate both ratings from pre-game snapshots.
P1: First Move Time Is Free
V1 problem:

White can wait indefinitely before first move.
V2 fix:

Start the clock when the game enters IN_PROGRESS.
P1: Timeout Requires A Move
V1 problem:

Timeout is checked only when a move is submitted.
V2 fix:

Allow server-side timeout checks through a clock service and timeout endpoint/job.
P1: Deposit Verification Not Bound To Player
V1 problem:

A player can submit another player's deposit transaction.
V2 fix:

Verify tx.from, role, amount, game/challenge id, chain, contract, and event logs.
P2: Private Games Joinable By ID
V1 problem:

Private games can be joined through the normal join route.
V2 fix:

Private games require invite token validation.
Direct join by id is rejected unless the invite is valid.

Guiding Principle
Keep the system layered:

Chess engine decides chess.
Game service decides game state.
Challenge service decides challenge eligibility.
Tournament service decides tournament progression.
Escrow service verifies funds.
Settlement service coordinates payouts.
Contracts enforce money movement.
Multisig governs final settlement authority.

If a piece of logic does not belong to a layer, move it out early. V2 should be easier to reason about than V1 because each layer has one job.
